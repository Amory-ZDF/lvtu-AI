# 旅图 AI 产品数据看板设计文档

> 版本：V1.0
> 产品：旅图 Lv - AI 旅行规划平台
> 统计时区：Asia/Shanghai
> 适用角色：产品、运营、研发、AI/算法团队

## 1. 文档目标

旅图的核心用户旅程是：灵感发现 → 输入旅行偏好 → AI 推荐目的地 → 选择目的地 → 生成并对比路线 → 创建行程 → 编辑行程 → 使用地图、天气、机位、穿搭、打包与协同功能 → 出行执行。

数据看板需要回答四个问题：

1. 用户从哪里来，在哪一步流失？
2. 用户是否完成了真正有价值的旅行规划，而不只是浏览页面？
3. AI 推荐和路线是否满足用户约束、可执行且稳定？
4. 哪些功能推动用户保存、回访、协同和实际使用行程？

## 2. 北极星指标与护栏指标

### 2.1 北极星指标：周有效规划用户数 WEP

**定义**：自然周内成功创建至少一个行程，并在创建后 7 天内发生至少一种后续价值行为的去重用户数。

后续价值行为包括：编辑或自动保存行程、自然语言调整行程、创建版本、查看地图/天气、查看机位/穿搭/打包、邀请协作者、再次进入该行程。

**公式**：`COUNT(DISTINCT user_key WHERE trip_created AND value_action_within_7d)`。

**含义**：避免把只访问首页、只生成推荐或创建后不再使用的用户误认为成功用户。它同时体现规划完成和持续使用价值。

### 2.2 护栏指标

| 指标 | 公式 | 含义 |
|---|---|---|
| 有效规划率 | 周有效规划用户 / 开始偏好输入用户 | 主链路真实转化质量 |
| 目的地推荐成功率 | 推荐成功请求 / 推荐发起请求 | AI 推荐基础可用性 |
| 路线生成成功率 | 路线成功请求 / 路线发起请求 | 核心规划稳定性 |
| 约束满足率 | 满足预算、天数、季节、兴趣等约束的评测 Case / 全部 Case | 防止以转化换低质量推荐 |
| 行程创建耗时 P90 | 从偏好提交到 `trip_created` 的 P90 时长 | 用户等待体验 |
| D7 有效规划留存 | 激活用户中第 7 日仍有价值行为的用户 / 激活用户 | 产品持续价值 |
| AI 单次有效规划成本 | AI/地图/生图等费用 / 有效规划数 | 单位经济性 |

## 3. 指标体系

数据状态：**已有**表示当前事件或业务表可算；**需补**表示需要新增事件、字段或聚合。

### 3.1 用户规模与获客

| 指标 | 口径 | 含义 | 状态 |
|---|---|---|---|
| UV | 按 `user_id > visitor_id > session_id` 去重 | 实际触达用户规模 | 已有 |
| 新访客数 | 首次出现的 `visitor_id/user_id` | 新增流量 | 需补首次访问维表 |
| 注册用户数 | 注册成功用户数 | 匿名到账号转化 | 业务表可算 |
| 登录用户率 | 登录用户 UV / 总 UV | 身份沉淀程度 | 已有 |
| 渠道 UV | 按 UTM/referrer 去重用户 | 获客来源质量 | 需规范属性 |
| 首页跳出率 | 单页且无有效交互会话 / 首页落地会话 | 首屏吸引力 | 已有 |

### 3.2 新用户激活与核心规划漏斗

主漏斗必须按同一用户、严格事件顺序、限定窗口计算：

`访问首页 → 进入偏好页 → 提交偏好 → 推荐成功 → 选择目的地 → 路线生成成功 → 确认路线 → 创建行程 → 有效规划`

| 指标 | 公式 | 含义 | 状态 |
|---|---|---|---|
| 首页到偏好页转化率 | 进入 `/start` 用户 / 首页用户 | CTA 与价值表达效果 | 已有 |
| 偏好提交率 | 提交偏好用户 / 进入偏好页用户 | 表单摩擦程度 | 需明确事件 |
| 推荐成功率 | 推荐成功 / 推荐发起 | AI 推荐稳定性 | 需补发起和失败 |
| 推荐选择率 | 选择目的地用户 / 推荐成功用户 | 推荐相关性与说服力 | 已有 |
| 路线生成成功率 | 路线成功 / 路线发起 | 路线服务稳定性 | 需补发起和失败 |
| 方案选择率 | 点击路线方案用户 / 查看对比页用户 | 方案吸引力 | 已有 |
| 方案确认率 | 确认方案用户 / 点击方案用户 | 决策确定性 | 已有 |
| 行程创建率 | 创建行程用户 / 推荐成功用户 | 完整规划转化 | 已有 |
| 激活率 | 有效规划用户 / 新用户 | 新用户达到核心价值比例 | 需派生 |
| 首次价值时间 TTFV | 首次访问到首次 `trip_created` 或有效规划 | 用户获得价值的速度 | 需有序事件 |

注意：地图、天气、穿搭、机位、打包和自动保存不是主漏斗的连续步骤，应作为行程创建后的功能采用漏斗独立计算。现有接口将这些步骤串成单漏斗，会导致相邻转化率失真。

### 3.6 留存与用户分群

| 指标 | 口径 | 含义 |
|---|---|---|
| D1/D7/D30 留存 | 首次有效规划 Cohort 在第 N 日有价值行为的用户占比 | 持续价值 |
| 行程周期留存 | 创建后、出发前、出行中、结束后各阶段活跃率 | 旅行低频产品更合理的留存口径 |
| 新老用户占比 | 首次访问/历史访问用户分布 | 用户结构 |
| 激活用户留存 | 仅对完成有效规划用户计算留存 | 排除低意向流量干扰 |
| 复购式规划率 | 90 天内创建第二个不同目的地行程的用户 / 创建用户 | 长周期复用价值 |

旅行产品天然低频，不能只用日活判断价值；应同时观察行程生命周期和 30/90 天复用。

### 3.7 AI 产品质量

结合现有评测体系，展示线上结果和离线评测两层指标。

| 指标 | 含义 | 来源 |
|---|---|---|
| 结构完整性分 | 输出字段能否被前端稳定消费 | 离线评测 |
| 场景相关性分 | 是否符合季节、兴趣、预算和人群 | 离线评测 |
| 约束满足度 | 是否遵守天数、预算和排除条件 | 离线评测 |
| 可执行性分 | 路线顺序、强度、交通和时间是否合理 | 离线评测 |
| 可解释性分 | 推荐理由是否具体可信 | 离线评测 |
| 安全体验分 | 是否包含风险、天气、高海拔等提醒 | 离线评测 |
| 平均总分/低分率 | AI 版本整体质量与低于 70 分 Case 占比 | 离线评测 |
| Badcase 归因 | 数据不足、检索失败、提示词、接口结构、产品策略占比 | 离线评测 |
| 线上采纳率 | 推荐或方案被选择用户 / 曝光用户 | 线上事件 |
| 生成后重试率 | 重新生成用户 / 生成成功用户 | 需补事件 |
| 生成后人工修改率 | 创建后发生编辑行程 / 创建行程 | 线上代理质量指标 |

线上采纳率受 UI、排序和用户意图影响，不能单独代表模型质量；必须与固定测试集评测结合。

### 3.8 稳定性、性能与成本

| 指标 | 公式 | 含义 |
|---|---|---|
| API 成功率 | 2xx 请求 / 全部请求 | 服务可用性 |
| 推荐/路线任务成功率 | completed jobs / started jobs | 核心 AI 稳定性 |
| P50/P90/P95 延迟 | 按接口与任务类型统计 | 典型与长尾等待 |
| 超时率 | 超时请求 / 发起请求 | 用户卡顿风险 |
| 错误码分布 | 各错误码 / 全部错误 | 故障定位 |
| Mock/真实 AI 占比 | 各 provider 请求 / 全部 AI 请求 | 防止把 Mock 表现当真实能力 |
| 单次生成成本 | 模型、地图、生图费用 / 成功生成次数 | 单次调用经济性 |
| 单个有效规划成本 | 全部可变成本 / 有效规划数 | 核心单位经济性 |

## 4. 统一口径

1. 业务日期统一使用北京时间，原始时间保留时区。
2. 用户去重优先级：`user_id`、`visitor_id`、`session_id`。用户登录后要建立 visitor 与 user 的身份合并关系，否则同一人会重复计算。
3. 漏斗按用户和事件顺序计算，并设置 7 天转化窗口；不能只统计周期内各事件 UV 后直接相除。
4. 主漏斗与功能采用漏斗分开；地图、天气、机位、穿搭、打包之间不存在强制先后顺序。
5. 按钮 CTR 当前是点击次数 / 页面 PV，不是按钮曝光 CTR。关键卡片需补 `element_exposure` 后再使用曝光作分母。
6. 动态路径 `/trips/{id}` 聚合为页面模板 `/trips/:id`，同时保留 `trip_id` 属性。
7. 过滤 `/data-center` 管理员行为、测试环境、自动化测试和内部账号。
8. 所有比例展示分子、分母和样本量；低样本只做观察，不直接决策。

## 5. 事件方案

### 5.1 公共属性

每个事件建议包含：`event_id`、`event_name`、`event_time`、`user_id`、`visitor_id`、`session_id`、`page_path`、`page_template`、`trip_id`、`destination_id/name`、`job_id`、`provider`、`model_version`、`app_version`、`environment`、`device_type`、`utm_source/medium/campaign`、`request_id`、`success`、`error_code`、`duration_ms`。

隐私要求：不采集输入框正文、密码、Token、API Key、身份证、精确住址或精确定位；自由文本如需分析，应先分类再上报标签。

### 5.2 保留的现有事件

`page_view`、`page_leave`、`page_heartbeat`、`button_click`、`link_click`、`form_submit`、`destination_recommendation_success`、`destination_selected`、`route_generation_success`、`route_option_selected`、`route_option_confirmed`、`trip_created`、`outfit_preview_generated`、`map_viewed`、`weather_loaded`、`trip_tab_view`、`itinerary_adjustment_success`、`trip_point_autosaved`。

### 5.3 必补事件

| 事件 | 触发时机 | 核心属性 |
|---|---|---|
| `preference_submitted` | 偏好校验通过并提交 | duration_days、budget_bucket、interest_count |
| `destination_recommendation_started/failed` | 推荐请求开始/失败 | job_id、provider、error_code、duration_ms |
| `destination_card_exposed` | 卡片有效曝光 | destination、rank、recommendation_id |
| `route_generation_started/failed` | 路线请求开始/失败 | destination、job_id、error_code |
| `route_option_exposed` | 方案有效曝光 | option_id、rank、pace、budget |
| `recommendation_regenerated` | 用户要求重新生成 | module、reason |
| `trip_edit_started/saved/failed` | 行程编辑生命周期 | trip_id、edit_type、changed_count |
| `trip_version_created/restored` | 创建或恢复快照 | trip_id、version_number |
| `collaboration_invited/accepted/edited` | 协同生命周期 | trip_id、role、invite_channel |
| `trip_shared` | 分享成功 | trip_id、channel |
| `feature_result_empty/failed` | 机位、穿搭、天气等为空或失败 | module、error_code |
| `ai_feedback_submitted` | 点赞、点踩或原因反馈 | module、rating、reason_code |

成功和失败必须由服务端上报；前端点击只能说明用户尝试，不能代表业务成功。

## 6. 常见业务事件看什么指标

| 问题 | 第一层指标 | 诊断顺序 |
|---|---|---|
| 首页流量下降 | UV、新访客、渠道 UV | 数据完整性 → 渠道 → 设备 → 新老用户 |
| 首页访问高但开始规划少 | 首页到偏好页转化、CTA 会话点击率、跳出率 | 首屏价值表达 → CTA 曝光/点击 → 加载与移动端 |
| 偏好页流失高 | 偏好提交率、停留分布、校验失败率 | 字段步骤 → 预算/天数 → 错误提示 → 设备 |
| 推荐成功率下降 | started/success/failed、P90、错误码 | Provider/模型 → 版本 → 输入分群 → 超时 |
| 推荐成功但不选择 | 卡片曝光、目的地选择率、位次转化 | 推荐相关性 → 图片/理由 → 供给覆盖 → 排序偏差 |
| 路线生成后不确认 | 方案曝光/选择/确认率、可执行性分 | 预算与节奏 → 方案差异 → 详情可读性 → AI 质量 |
| 行程创建后不再使用 | 7 日再访问、编辑率、功能采用率 | 行程完整性 → 保存反馈 → 提醒触达 → 出发日期分群 |
| 地图/机位/穿搭采用低 | 模块曝光、点击、成功率、空结果率 | 入口发现 → 数据覆盖 → 生成质量 → 场景相关性 |
| AI Badcase 增加 | 低分率、六维评分、归因分布 | 数据不足 → 检索 → Prompt → 接口结构 → 产品策略 |
| D7 留存下降 | 激活 Cohort 留存、价值行为构成 | 获客质量 → 首次体验 → 行程完整度 → 生命周期阶段 |
| 协同低使用 | 邀请率、接受率、协同行程率 | 入口 → 邀请到达 → 权限 → 实时连接稳定性 |
| 成本上涨 | 单次成本、重试率、有效规划成本 | 调用量 → 单价 → 失败/重试 → 模型版本 → 缓存命中 |

固定分析顺序：确认埋点完整 → 找漏斗断点 → 拆用户/渠道/设备 → 看性能错误 → 看 AI 质量 → 对照版本和运营变化。

## 7. 看板信息架构

### 页面 1：经营总览

首屏卡片：周有效规划用户、有效规划率、新用户激活率、D7 留存、目的地推荐成功率、路线生成成功率、有效规划成本。展示当前值、环比、目标差、样本量。

第二屏：7/30/90 日趋势、严格有序主漏斗、异常告警。第三屏：目的地与方案 Top、AI 低分率、需关注错误明细。

### 页面 2：核心旅程漏斗

按用户 Cohort 展示主漏斗，支持新/老、登录/游客、渠道、设备、预算、天数、兴趣、AI Provider 和版本筛选。展示步骤耗时与流失用户，不把后置功能混入主漏斗。

### 页面 3：页面与交互

保留现有页面停留分位数、跳出/退出、按钮点击和选择分布。补充页面趋势、关键元素曝光 CTR、动态路径归一化、前后路径和空状态/错误状态分析。

### 页面 4：行程使用与协同

展示创建、编辑、调整、版本、地图、天气、机位、穿搭、打包、再访问和临近出行活跃；协同展示邀请 → 接受 → 首次共同编辑漏斗。

### 页面 5：留存与用户分群

展示新用户 D1/D7/D30 留存、激活用户留存、行程生命周期留存和 90 天再次规划率。支持按首次选择目的地、预算、兴趣、渠道和设备分群。

### 页面 6：AI 质量

线上展示推荐采纳、重试、人工修改、反馈、空结果；离线展示六维评分、平均分、低分率、Badcase 归因和版本对比。每次发布前后展示同一测试集差异。

### 页面 7：稳定性与成本

按接口、任务类型、Provider、模型和版本展示成功率、P50/P90/P95、超时、错误码、重试、Mock/真实占比、单次成本和有效规划成本；可下钻 `request_id/job_id`。

### 页面 8：数据质量

展示事件延迟、重复率、关键事件缺失率、匿名身份合并率、事件顺序异常、动态路径未归一化、测试流量占比。数据不可信时对应业务指标显示警告。

## 8. 现有能力审计

### 8.1 可直接复用

- `analytics_events` 已包含用户、访客、会话、事件、页面、元素、停留、设备和 JSON metadata。
- `/data-center` 已支持基础漏斗、页面停留分位数、按钮点击、事件分类和偏好选择分布。
- `trips`、`plan_variants`、`trip_versions`、`collaborators`、`generation_jobs` 等业务表可用于校验核心结果。
- 现有 AI 评测脚本和报告可作为离线质量数据源。

### 8.2 必须修正

1. 当前 dashboard 以周期内事件 UV 集合计算漏斗，未校验同一用户的严格先后顺序，可能出现转化率失真。
2. 当前把行程详情的可选功能串成连续漏斗，应拆为独立采用率。
3. `page_path` 包含查询参数和动态行程 ID，需要增加 `page_template`。
4. 点击率以页面 PV 为分母不是元素曝光 CTR，关键推荐卡和 CTA 需补曝光。
5. 只有成功事件不足以计算成功率，需补 started/failed。
6. 聚合层未结合业务表，`trip_created` 等关键结果应以业务表为准、事件表为行为解释。

## 9. 数据建设方案

当前阶段不需要立刻引入重型数仓。建议：

1. PostgreSQL 继续存业务表与原始事件表。
2. 新增身份映射表，关联匿名 visitor 与登录 user。
3. 增加服务端业务事件、失败事件和任务成本字段。
4. 建立日级聚合表：用户日活、漏斗 Cohort、功能采用、留存、AI 质量、任务性能。
5. dashboard 接口读取聚合结果，避免每次全量加载事件到 Python 内存。
6. 数据量明显增长后再评估 ClickHouse/Doris 或成熟 BI。

离线评测结果建议标准化为：`evaluation_run`、`evaluation_case_result`、`model_version`、六维得分、总分、主归因、运行时间，并与发布版本关联。

## 10. 分阶段实施

### P0：修正口径与可观测性

- 拆分主漏斗和功能采用漏斗。
- 增加 started/failed、偏好提交、曝光、编辑/版本事件。
- 规范 `page_template`、`trip_id`、Provider、模型版本、环境和错误码。
- 业务表校验行程创建、协同和版本结果。
- 验收：可还原单个用户完整有序旅程；核心成功率有正确分母。

### P1：经营、留存与 AI 质量

- 上线北极星、趋势、Cohort 留存、行程生命周期、分群。
- 接入评测报告和 Badcase 归因。
- 增加模型版本、任务耗时、Mock/真实 AI 和成本。
- 验收：能回答转化为何变化、用户为何回访、AI 哪一维退化。

### P2：协同、反馈与实验

- 完善邀请、接受、共同编辑、分享和用户反馈闭环。
- 增加实验 ID、实验组、曝光与目标指标，支持 A/B 测试。
- 建立异常告警和自动周报。
- 验收：能评估一次产品、Prompt、模型或数据升级是否真正提升有效规划。

## 11. 刷新、告警与使用机制

- 核心错误与任务状态：5 分钟；经营指标：每小时；留存和 AI 评测：每日或每次版本发布。
- 无历史基线时先观察 4 周。临时告警：核心任务成功率低于 95%、P90 超近 7 日中位数 2 倍、关键事件缺失超过 5%、D7 留存或有效规划率连续两周期下降 20%。
- 每日运营看异常与主漏斗；每周产品看转化、留存和功能采用；每次 AI 发布看固定集评分与 Badcase；每月看有效规划成本和复用率。

## 12. 验收清单

- [ ] 北极星有明确用户、窗口和价值行为定义。
- [ ] 主漏斗严格按用户、顺序和窗口计算。
- [ ] 主漏斗与后置功能采用已拆分。
- [ ] 成功率同时有 started、success、failed。
- [ ] 关键卡片和方案有曝光事件，CTR 分母正确。
- [ ] 匿名到登录用户可以合并。
- [ ] 动态路径归一化，测试和管理员流量已过滤。
- [ ] 线上行为与离线 AI 评测同时展示。
- [ ] 指标可下钻到用户旅程、trip、job 和 request。
- [ ] 所有指标显示公式、样本量、更新时间和数据源。

## 13. 最终建议

旅图当前不缺“更多事件列表”，而是缺少一套围绕有效规划的经营口径。第一优先级应是修正有序漏斗、拆分后置功能、补失败和曝光事件，并用业务表校验行程结果；第二优先级是建立激活与留存；第三优先级才是协同、实验和成本优化。这样数据中台才能从行为陈列页升级为真正支持产品决策的看板。
