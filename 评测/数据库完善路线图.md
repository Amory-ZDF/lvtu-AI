# 旅图数据库完善路线图

## 1. 当前问题判断

从产品角度看，数据库太小会导致三个直接问题：

- 推荐结果重复：用户换一个条件，仍然出现相似目的地和路线。
- 场景无法细分：亲子、情侣、摄影、徒步、低预算、轻奢等需求难以被区分。
- AI 输出泛化：模型会用常识补内容，导致看起来合理但不够真实、不可执行。

## 2. 数据库应该服务哪些产品能力

旅图的数据库不应该只是“地点列表”，而应该支撑四类决策：

- **去哪里**：目的地库 `destinations`
- **怎么玩**：旅行笔记 / 路线经验库 `travel_notes`
- **哪里拍**：机位库 `photo_spots`
- **怎么穿**：穿搭库 `outfits`

## 3. 建议数据量目标

### 第一阶段：能 Demo 稳定

- 目的地：50-80 个
- 每个重点目的地机位：8-15 个
- 穿搭规则：按季节 × 场景至少 80-120 条
- 旅行笔记：300-500 条清洗后的高质量内容

### 第二阶段：能灰度给真实用户

- 目的地：150-250 个
- 机位：1000+ 个
- 穿搭：500+ 条
- 旅行笔记：3000-5000 条

### 第三阶段：做成产品壁垒

- 为每个目的地建立结构化画像：预算、季节、交通、适合人群、风险、关键词。
- 为每类用户建立偏好画像：摄影、美食、亲子、情侣、特种兵、疗愈、户外。
- 为每次 AI 生成结果记录命中数据和用户反馈。

## 5. 推荐扩充路径

### Step 1：先做 20 个标杆目的地

不要一开始追求全国覆盖，先选 20 个高频目的地，每个目的地做深。

建议优先：

- 北京、上海、杭州、苏州、南京、成都、重庆、西安、厦门、广州
- 大理、丽江、三亚、青岛、长沙、武汉、桂林、敦煌、拉萨、香港

每个目的地至少补齐：

- 10 条机位
- 5 条穿搭场景
- 10 条旅行笔记
- 2 条典型路线：慢节奏 / 高效打卡

### Step 2：建立数据入库流水线

当前项目已有基础脚本：

```bash
cd backend
LV_SEED_DATA_DIR="/path/to/lv_private_data/seed_data" python -m scripts.seed_knowledge --reset
python -m scripts.crawl_xiaohongshu --keywords 厦门攻略,大理攻略 --limit 20
python -m scripts.clean_and_import --dry-run
python -m scripts.clean_and_import
```

建议后续补一个数据质量报告脚本，统计：

- 每个目的地有多少机位。
- 每个目的地有多少旅行笔记。
- 哪些季节没有穿搭。
- 哪些数据缺少经纬度或图片。

### Step 3：用评测结果反推补数据

如果某条测试用例低分：

- 目的地推荐泛泛：补 `destinations`。
- 行程不可执行：补 `travel_notes` 和结构化路线经验。
- 机位空泛：补 `photo_spots` 的构图、时间、设备字段。
- 穿搭不实用：补 `outfits` 的季节、天气、场景字段。

### Step 4：建立数据审核标准

每条入库数据至少过四个检查：

- 是否真实：不要编造不存在的地点或机位。
- 是否可执行：用户能否按地址、时间、注意事项行动。
- 是否结构化：字段齐全，能被前端直接展示。
- 是否可维护：来源、更新时间、目的地标签明确。

## 6. 隐私和合规提醒

- `.env`、API Key、Token、真实用户数据不能入库上传 GitHub。
- 原始抓取数据建议放在被 `.gitignore` 忽略的目录，例如 `backend/app/data/raw_notes/` 或本地临时目录。
- 上传 GitHub 前必须执行：

```bash
git status --short
git ls-files | grep -E '(^|/)\.env|secret|token|key'
```

如果出现真实 `.env` 或私钥文件，必须停止提交。
